## 8. 常见问题

### 8.1 `torch.cuda.is_available()` 为 False

优先确认 `rtdetr_5070ti` 环境里的 PyTorch 是否支持当前 GPU 和 CUDA。不要先改代码。

```powershell
nvidia-smi
python -c "import torch; print(torch.__version__); print(torch.version.cuda); print(torch.cuda.is_available())"
```

### 8.2 AP 明显低于官方结果

最常见原因是 COCO 类别映射或数据路径不一致。确认：

```yaml
num_classes: 80
remap_mscoco_category: True
```

### 8.3 权重加载报 key 不匹配

评估官方完整权重时使用 `-r`；基于预训练权重微调时使用 `-t`。不要混用：

```powershell
# 评估或恢复训练
python tools/train.py -c configs/rtdetr/rtdetr_r18vd_6x_coco.yml -r path\to\checkpoint.pth --test-only

# 微调，只加载可匹配模型参数
python tools/train.py -c configs/rtdetr/rtdetr_r18vd_6x_coco.yml -t path\to\checkpoint.pth
```

### 8.4 单卡显存不足

R18 默认训练 batch size 为 4。若显存不足，可临时把 `rtdetr_pytorch/configs/rtdetr/include/dataloader.yml` 的 `train_dataloader.batch_size` 改小，但严格复现实验应记录这个改动，因为 batch size 会影响结果。

### 8.5 训练启动后 `SanitizeBoundingBox` 报错

若训练在构建 `train_dataloader` 时退出，并在 `console_train.err.log` 中看到：

```text
KeyError: 'SanitizeBoundingBox'
```

原因是官方配置使用旧 torchvision v2 beta 名称 `SanitizeBoundingBox`，而当前环境是 torchvision 0.23，对应实现名为 `SanitizeBoundingBoxes`。当前项目已在 `rtdetr_pytorch/src/data/transforms.py` 中注册旧名称兼容类。

若继续出现：

```text
TypeError: _DataPointsCompat.BoundingBox() missing ... 'format' and 'spatial_size'
```

原因是旧 `datapoints.BoundingBox` 在 torchvision 0.23 中已迁移到 `tv_tensors.BoundingBoxes`，并且需要既能构造对象，也能作为类型参与 transform 类型检查。当前项目已在 `rtdetr_pytorch/src/data/datapoints_compat.py` 中补齐类型兼容。

已验证命令：

```powershell
python -m unittest tests.test_transforms_compat -v
python -m unittest tests.test_training_dashboard -v
```

并已用单进程 dataloader 探针验证训练集能取到一批 `(4, 3, 640, 640)` 样本。

### 8.6 第一轮 loss 极大或 AP 为 0

若 epoch 0 出现类似 `train_loss` 约 45000 且 `AP=0.0`，这是异常结果，不应继续作为正式复现记录。已定位到 torchvision 0.23 兼容层问题：`Resize` 后的 boxes 变成 `torchvision.tv_tensors.BoundingBoxes` 基类，原 `ConvertBox` 只识别旧兼容子类，导致 box 没有转换为归一化 `cxcywh`，仍以 0-640 像素坐标进入损失函数。

当前已修复 `rtdetr_pytorch/src/data/transforms.py` 和 `rtdetr_pytorch/src/data/datapoints_compat.py`，并新增测试覆盖 `ConvertBox` 和 `PadToSize` 对 torchvision 0.23 `BoundingBoxes` 的处理。

异常 run 已归档到：

```text
rtdetr_pytorch/output/invalid_runs/r18_bbox_not_normalized_2026-05-18/
```

归档内容包括错误 run 的 `log.txt`、`console_train.log`、`console_train.err.log`、`checkpoint.pth`、`checkpoint0000.pth` 和探针日志。这组结果不要用于论文复现对比，也不要从该 checkpoint resume。

修复后已验证真实 COCO 训练 batch 的 `target["boxes"]` 为 0-1 范围内的归一化张量。重新正式训练时，epoch 0 应参考官方日志：`train_loss` 约 15.46，`AP` 约 0.1605。


### 加速建议

对单张 RTX 5070 Ti Laptop GPU，COCO 上 RT-DETR-R18 每个 epoch 约 1.7-2 小时属于正常量级。为了尽量加速，同时不明显破坏论文复现可比性，建议按以下顺序尝试：

1. 优先尝试开启 `AMP`。它通常能降低显存占用并提高吞吐，也可能缓解 epoch 7 出现的 CUDA OOM。严格复现记录里需要注明 `AMP=True`，因为最终 AP 可能有轻微差异。
2. `num_workers` 从 `4` 小步试到 `6` 或 `8`。Windows 上 workers 过多可能不稳定，所以建议一次只改一个值，观察首 batch 等待时间、GPU 利用率和是否报错。
3. 不建议增大 `训练 batch size`。当前 batch size 4 已经在 epoch 7 出现过 OOM，再增大风险更高。
4. 如果清理显存和 AMP 后仍 OOM，可把 `训练 batch size` 临时降到 `3` 或 `2`，但这会改变有效 batch size，严格复现时必须记录。
5. 不建议为了提速修改输入尺寸、模型结构、epoch 数或学习率；这些会直接改变论文结果可比性。
